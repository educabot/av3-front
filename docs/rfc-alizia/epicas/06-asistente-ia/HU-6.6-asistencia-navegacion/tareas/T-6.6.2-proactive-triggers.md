# T-6.6.2: Sistema de triggers proactivos

**Historia:** HU-6.6 — Asistencia de navegacion
**Tipo:** Backend + Frontend
**Estado:** ⬜ Pendiente
**Fase:** Post-MVP

---

## Descripcion

Implementar la logica que detecta situaciones donde Alizia debe ofrecer ayuda proactivamente: primera visita a una pantalla, inactividad prolongada, pantalla vacia, o documento recien publicado. Los triggers se evaluan en el frontend y el mensaje proactivo se genera en el backend.

## Triggers definidos

| Trigger | Condicion | Mensaje tipo |
|---------|-----------|--------------|
| `first_visit` | Primera vez que el usuario visita una pantalla clave | "Bienvenido a [pantalla]. Puedo explicarte como funciona." |
| `inactivity` | Usuario lleva >60s en un wizard sin avanzar | "¿Necesitas ayuda con este paso?" |
| `empty_screen` | Pantalla de planificacion sin contenido | "Para empezar, selecciona un nucleo problematico..." |
| `published_doc` | Usuario abre un documento recien publicado | "Este documento cubre X clases y Y categorias. ¿Queres un resumen?" |

## Implementacion

### Frontend: evaluacion de triggers

```javascript
// El frontend detecta la condicion y solicita un mensaje proactivo al backend
const TRIGGER_CONFIGS = {
    first_visit: {
        check: (screen) => !localStorage.getItem(`visited_${screen}`),
        onTrigger: (screen) => localStorage.setItem(`visited_${screen}`, 'true')
    },
    inactivity: {
        check: (screen, elapsed) => elapsed > 60000 && WIZARD_SCREENS.includes(screen),
        cooldown: 300000 // no repetir por 5 minutos
    },
    empty_screen: {
        check: (screen, _, data) => PLANNABLE_SCREENS.includes(screen) && !data.hasContent
    },
    published_doc: {
        check: (screen, _, data) => screen === 'document_view' && data.status === 'published' && data.justOpened
    }
};
```

### Backend: endpoint de mensaje proactivo

```
POST /alizia/proactive-message
```

```json
{
    "trigger": "first_visit",
    "navigation_context": {
        "screen": "wizard_step_1",
        "route": "/courses/3/documents/new/step-1",
        "visible_data": { "course_id": 3 }
    }
}
```

Respuesta:
```json
{
    "message": "¡Bienvenido al wizard de creacion! Aca vas a seleccionar los nucleos problematicos y categorias que queres trabajar en el documento. ¿Te explico que es cada uno?",
    "dismissable": true,
    "actions": [
        { "label": "Explicame", "action": "explain" },
        { "label": "Ya se como funciona", "action": "dismiss" }
    ]
}
```

## Comportamiento

1. El frontend evalua los triggers en cada cambio de pantalla y periodicamente (para inactivity)
2. Si un trigger se activa, envia request al backend con el trigger type y navigation context
3. El backend genera un mensaje contextual usando el AI Provider (o un template pre-escrito para triggers frecuentes)
4. El frontend muestra el mensaje en el widget de chat con opciones de accion
5. Si el usuario hace dismiss, el trigger no se repite (respeta cooldown o `first_visit` flag)

## Notas

- **Principio: lo proactivo debe ser infrecuente y dismissable** — el usuario siempre puede cerrar
- Los triggers pre-escritos (first_visit, inactivity) no necesitan llamada al LLM — solo los contextuales (published_doc summary)
- El cooldown previene que Alizia se vuelva molesta repitiendo sugerencias
- La feature `recommendations` en ai_settings (HU-6.5) controla si los proactivos estan habilitados
- User testing es necesario para calibrar los umbrales (60s de inactividad, 5min de cooldown)
