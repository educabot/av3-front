## Descripcion

Tests unitarios para el AI Provider. Se mockea el cliente de Azure OpenAI (o se usa un HTTP server mock) para no depender del servicio real en CI. Se cubren los 3 metodos, el retry con backoff y la clasificacion de errores.

### Mock del cliente

```go
// tests/ai/mock_client.go
package ai_test

import (
    "github.com/Azure/azure-sdk-for-go/sdk/ai/azopenai"
)

// MockAzureClient permite controlar las respuestas del SDK para tests.
type MockAzureClient struct {
    Responses []azopenai.GetChatCompletionsResponse
    Errors    []error
    CallCount int
}

func (m *MockAzureClient) GetChatCompletions(
    ctx context.Context,
    body azopenai.ChatCompletionsOptions,
    options *azopenai.GetChatCompletionsClientOptions,
) (azopenai.GetChatCompletionsResponse, error) {
    idx := m.CallCount
    m.CallCount++
    if idx < len(m.Errors) && m.Errors[idx] != nil {
        return azopenai.GetChatCompletionsResponse{}, m.Errors[idx]
    }
    if idx < len(m.Responses) {
        return m.Responses[idx], nil
    }
    return azopenai.GetChatCompletionsResponse{}, fmt.Errorf("no more mock responses")
}
```

## Notas

- NO se hacen llamadas reales a Azure OpenAI en tests unitarios
- Para poder inyectar el mock, considerar extraer una interfaz interna del cliente Azure o usar un wrapper
- Los tests de retry deben verificar el numero de intentos (via `CallCount`) sin esperar el backoff real — usar un clock mock o reducir los tiempos en tests
- Los tests de `classifyError` son criticos: cada tipo de error de Azure debe mapearse al error tipado correcto
- Considerar tests de integracion (opcionales, tag `//go:build integration`) que si llamen a Azure para validar la configuracion en staging
