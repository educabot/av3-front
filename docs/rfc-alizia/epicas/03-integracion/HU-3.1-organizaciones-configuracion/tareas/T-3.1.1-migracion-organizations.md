# T-3.1.1: Migración: organizations

**Historia:** HU-3.1 — Organizaciones y configuración
**Tipo:** DB
**Estado:** ⬜ Pendiente

---

## Descripción

Verificar que la tabla `organizations` existe (creada en T-1.2.1 de la Épica 1) y agregar columnas o índices faltantes si es necesario. La tabla base ya fue creada en la migración de HU-1.2 con los campos: id, name, slug (unique), config (JSONB), created_at. Esta tarea se asegura de que el esquema esté completo para los requerimientos de configuración organizacional.

## Implementación

### SQL de verificación / migración incremental

Si la tabla ya existe desde T-1.2.1, no se necesita crear de nuevo. Solo verificar y agregar lo faltante:

```sql
-- Verificar que la tabla existe (referencia desde T-1.2.1)
-- CREATE TABLE organizations (
--     id SERIAL PRIMARY KEY,
--     name VARCHAR(255) NOT NULL,
--     slug VARCHAR(100) NOT NULL UNIQUE,
--     config JSONB NOT NULL DEFAULT '{}',
--     created_at TIMESTAMP NOT NULL DEFAULT NOW()
-- );

-- Índice GIN sobre config para queries eficientes sobre JSONB
CREATE INDEX IF NOT EXISTS idx_organizations_config ON organizations USING GIN (config);

-- Índice sobre slug para lookups rápidos (ya existe como UNIQUE, pero explícito)
CREATE INDEX IF NOT EXISTS idx_organizations_slug ON organizations(slug);
```

### SQL (si la tabla NO existe)

En caso de que la Épica 1 no se haya ejecutado previamente:

```sql
CREATE TABLE organizations (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    slug VARCHAR(100) NOT NULL UNIQUE,
    config JSONB NOT NULL DEFAULT '{}',
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_organizations_config ON organizations USING GIN (config);
CREATE INDEX idx_organizations_slug ON organizations(slug);
```

## Notas

- La tabla `organizations` fue definida originalmente en T-1.2.1 (Épica 1). Esta tarea es idempotente.
- El índice GIN sobre `config` es importante para queries que filtren por keys del JSONB (ej: `config->>'shared_classes_enabled'`).
- Usar `CREATE INDEX IF NOT EXISTS` para evitar errores si se ejecuta más de una vez.
- Si se necesita agregar columnas nuevas en el futuro (ej: `updated_at`), hacerlo en una migración separada.
