# T-3.3.1: Migración: tabla topics con auto-referencia

**Historia:** HU-3.3 — Topics — jerarquía curricular
**Tipo:** DB
**Estado:** ⬜ Pendiente

---

## Descripción

Crear la migración para la tabla `topics`, que modela la jerarquía curricular completa (núcleos problemáticos > áreas de conocimiento > categorías) mediante una estructura de árbol con auto-referencia (`parent_id`). Esto reemplaza las tres tablas separadas del modelo actual (`problematic_nuclei`, `knowledge_areas`, `categories`) por una sola tabla flexible con niveles dinámicos.

## SQL (up)

```sql
-- Topics (hierarchical curriculum taxonomy)
CREATE TABLE topics (
    id SERIAL PRIMARY KEY,
    organization_id INTEGER NOT NULL REFERENCES organizations(id),
    parent_id INTEGER REFERENCES topics(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    level INTEGER NOT NULL DEFAULT 1,
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_topics_organization_id ON topics(organization_id);
CREATE INDEX idx_topics_parent_id ON topics(parent_id);
CREATE INDEX idx_topics_level ON topics(level);
```

## SQL (down)

```sql
DROP TABLE IF EXISTS topics;
```

## Notas

- `parent_id` es nullable: los topics raíz (nivel 1) tienen `parent_id IS NULL`
- `ON DELETE CASCADE` en `parent_id` asegura que al borrar un nodo padre se eliminen todos sus hijos recursivamente
- El campo `level` es calculado (parent.level + 1) pero se persiste para facilitar queries de filtrado por nivel sin necesidad de joins recursivos
- El índice en `level` permite filtrar eficientemente por nivel (ej: "dame todos los de nivel 3")
- La cantidad máxima de niveles se controla desde `organizations.config.topic_max_levels` (ver T-3.3.4)
