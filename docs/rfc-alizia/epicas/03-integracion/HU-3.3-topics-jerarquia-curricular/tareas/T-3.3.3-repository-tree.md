# T-3.3.3: Repository GORM con armado de árbol en memoria

## Descripción

Implementar el repositorio GORM para `Topic` con la lógica de armado de árbol. Se usa el enfoque de query plano + ensamblado en memoria, que es más simple y suficiente para el volumen esperado (<1000 topics por organización). Se descarta el CTE recursivo para el MVP por complejidad innecesaria.

## Implementación

### Enfoque: Flat query + in-memory tree assembly

```go
package repositories

import (
    "alizia/src/core/entities"
    "gorm.io/gorm"
)

type TopicRepository struct {
    db *gorm.DB
}

func NewTopicRepository(db *gorm.DB) *TopicRepository {
    return &TopicRepository{db: db}
}

func (r *TopicRepository) ListTree(organizationID int) ([]entities.Topic, error) {
    var topics []entities.Topic
    err := r.db.
        Where("organization_id = ?", organizationID).
        Order("level ASC, name ASC").
        Find(&topics).Error
    if err != nil {
        return nil, err
    }
    return buildTree(topics), nil
}

// buildTree ensambla la estructura jerárquica en memoria
func buildTree(flat []entities.Topic) []entities.Topic {
    // Mapa de ID → puntero al topic
    byID := make(map[int]*entities.Topic, len(flat))
    for i := range flat {
        flat[i].Children = []entities.Topic{}
        byID[flat[i].ID] = &flat[i]
    }

    // Agrupar hijos bajo su padre
    var roots []entities.Topic
    for i := range flat {
        if flat[i].ParentID == nil {
            roots = append(roots, flat[i])
        } else {
            parent, ok := byID[*flat[i].ParentID]
            if ok {
                parent.Children = append(parent.Children, flat[i])
            }
        }
    }

    // Copiar children actualizados a roots
    for i := range roots {
        if updated, ok := byID[roots[i].ID]; ok {
            roots[i] = *updated
        }
    }

    return roots
}

func (r *TopicRepository) ListByLevel(organizationID int, level int) ([]entities.Topic, error) {
    var topics []entities.Topic
    err := r.db.
        Where("organization_id = ? AND level = ?", organizationID, level).
        Order("name ASC").
        Find(&topics).Error
    return topics, err
}

func (r *TopicRepository) GetChildren(parentID int) ([]entities.Topic, error) {
    var children []entities.Topic
    err := r.db.
        Where("parent_id = ?", parentID).
        Order("name ASC").
        Find(&children).Error
    return children, err
}
```

### Alternativa descartada: CTE recursivo

```sql
-- Para referencia futura si el volumen crece significativamente
WITH RECURSIVE topic_tree AS (
    SELECT *, ARRAY[id] AS path
    FROM topics
    WHERE parent_id IS NULL AND organization_id = $1
    UNION ALL
    SELECT t.*, tt.path || t.id
    FROM topics t
    JOIN topic_tree tt ON t.parent_id = tt.id
)
SELECT * FROM topic_tree ORDER BY path;
```

## Notas

- El `Order("level ASC, name ASC")` garantiza que los padres se procesan antes que los hijos en `buildTree`
- Para <1000 topics por org, el enfoque en memoria es O(n) en tiempo y espacio — más que suficiente
- `buildTree` es una función pura (sin side effects) lo que facilita los tests unitarios
- Se inicializa `Children` como slice vacío (`[]entities.Topic{}`) en lugar de nil para que el JSON serialice como `[]` en vez de `null`
- El CTE recursivo queda documentado como alternativa si en el futuro alguna org tiene volúmenes muy grandes
