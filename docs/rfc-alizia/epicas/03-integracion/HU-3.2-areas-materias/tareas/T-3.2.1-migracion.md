# T-3.2.1: Migración: areas, subjects, area_coordinators

**Historia:** HU-3.2 — Áreas y disciplinas
**Tipo:** DB
**Estado:** ⬜ Pendiente

---

## Descripción

Crear la migración con las tablas de áreas, disciplinas y la tabla intermedia para asignar coordinadores a áreas. Estas tablas dependen de `organizations` (HU-3.1) y `users` (HU-1.2).

## SQL (up)

```sql
-- Areas (agrupación de disciplinas)
CREATE TABLE areas (
    id SERIAL PRIMARY KEY,
    organization_id INTEGER NOT NULL REFERENCES organizations(id),
    name VARCHAR(255) NOT NULL,
    description TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Subjects (disciplinas pertenecientes a un área)
CREATE TABLE subjects (
    id SERIAL PRIMARY KEY,
    organization_id INTEGER NOT NULL REFERENCES organizations(id),
    area_id INTEGER NOT NULL REFERENCES areas(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

-- Area coordinators (M2M: un coordinador puede coordinar múltiples áreas)
CREATE TABLE area_coordinators (
    id SERIAL PRIMARY KEY,
    area_id INTEGER NOT NULL REFERENCES areas(id) ON DELETE CASCADE,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE(area_id, user_id)
);

-- Indexes
CREATE INDEX idx_areas_organization_id ON areas(organization_id);
CREATE INDEX idx_subjects_organization_id ON subjects(organization_id);
CREATE INDEX idx_subjects_area_id ON subjects(area_id);
CREATE INDEX idx_area_coordinators_area_id ON area_coordinators(area_id);
CREATE INDEX idx_area_coordinators_user_id ON area_coordinators(user_id);
```

## SQL (down)

```sql
DROP TABLE IF EXISTS area_coordinators;
DROP TABLE IF EXISTS subjects;
DROP TABLE IF EXISTS areas;
```

## Notas

- `description` es nullable en ambas tablas (no siempre se requiere)
- `ON DELETE CASCADE` en `subjects.area_id`: si se borra un área, se borran sus disciplinas
- `ON DELETE CASCADE` en `area_coordinators`: si se borra un área o usuario, se limpia la asignación
- `UNIQUE(area_id, user_id)` previene asignar el mismo coordinador dos veces a la misma área
- `organization_id` se replica en `subjects` para simplificar queries de multi-tenancy (evita JOIN con areas en cada query)
