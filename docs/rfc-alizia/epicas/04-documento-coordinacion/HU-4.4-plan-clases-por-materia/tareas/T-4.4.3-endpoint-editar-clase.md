# T-4.4.3: Endpoint para editar clase individual

**Historia:** HU-4.4 — Plan de clases por disciplina
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Implementar el endpoint `PATCH /api/v1/coord-doc-classes/:id` que permite al coordinador editar el titulo, objetivo y topics de una clase individual dentro del plan de clases de un documento de coordinacion. Solo se puede editar si el documento esta en estado borrador.

## Implementacion

### Endpoint

```
PATCH /api/v1/coord-doc-classes/:id
```

### Request body

```json
{
  "title": "Nuevo titulo de la clase",
  "objective": "Nuevo objetivo pedagogico",
  "topic_ids": [1, 3, 5]
}
```

Todos los campos son opcionales. Solo se actualizan los campos enviados.

### Validaciones

1. **La clase existe**: verificar que `coord_doc_classes` con el ID dado existe.
2. **Pertenece a la organizacion del usuario**: la clase pertenece a un documento cuyo area pertenece a la organizacion del usuario autenticado.
3. **Documento en edicion**: el documento asociado debe tener `status = 'in_progress'`. Si no, retornar `403 Forbidden`.
4. **topic_ids validos**: cada ID en `topic_ids` debe existir en `coord_doc_subject_topics` para la misma disciplina del documento. Si algun ID es invalido, retornar `422 Unprocessable Entity` con detalle.

### Logica de actualizacion de topics

Cuando se envian `topic_ids`, se reemplazan **todos** los registros de `coord_doc_class_topics` para esa clase:

```go
func (h *Handler) PatchCoordDocClass(c echo.Context) error {
    classID := c.Param("id")

    var req PatchClassRequest
    if err := c.Bind(&req); err != nil {
        return echo.NewHTTPError(400, "invalid request body")
    }

    // Validar pertenencia y estado
    class, err := h.classRepo.GetWithDocument(ctx, classID)
    if err != nil {
        return echo.NewHTTPError(404, "class not found")
    }

    if class.Document.Status != "in_progress" {
        return echo.NewHTTPError(403, "cannot edit classes of a non-in_progress document")
    }

    // Validar topic_ids si se enviaron
    if req.TopicIDs != nil {
        valid, err := h.subjectTopicRepo.ValidateIDs(ctx, class.DocSubjectID, req.TopicIDs)
        if !valid {
            return echo.NewHTTPError(422, "invalid topic_ids: some do not belong to this subject")
        }
    }

    // Actualizar en transaccion
    err = h.txManager.WithTransaction(ctx, func(tx Transaction) error {
        if req.Title != nil {
            class.Title = *req.Title
        }
        if req.Objective != nil {
            class.Objective = *req.Objective
        }
        if err := h.classRepo.Update(ctx, tx, class); err != nil {
            return err
        }

        if req.TopicIDs != nil {
            // Reemplazar topics de la clase
            if err := h.classTopicRepo.DeleteByClassID(ctx, tx, classID); err != nil {
                return err
            }
            for _, tid := range *req.TopicIDs {
                if err := h.classTopicRepo.Create(ctx, tx, classID, tid); err != nil {
                    return err
                }
            }
        }
        return nil
    })

    if err != nil {
        return echo.NewHTTPError(500, "failed to update class")
    }

    return c.JSON(200, class)
}
```

### Response exitosa (200)

```json
{
  "id": 42,
  "class_number": 3,
  "title": "Nuevo titulo de la clase",
  "objective": "Nuevo objetivo pedagogico",
  "topic_ids": [1, 3, 5]
}
```

### Registro en router

Agregar en `mapping.go`:

```go
coordClasses := v1.Group("/coord-doc-classes")
coordClasses.PATCH("/:id", handler.PatchCoordDocClass, RequireRole("coordinator"))
```

## Notas

- El reemplazo completo de `coord_doc_class_topics` es mas simple y predecible que un diff parcial.
- La validacion de `topic_ids` contra `coord_doc_subject_topics` asegura que solo se asignen topics que fueron distribuidos a esa disciplina en el documento.
- Considerar emitir un warning (no error) si la edicion deja topics del documento sin asignar a ninguna clase.
