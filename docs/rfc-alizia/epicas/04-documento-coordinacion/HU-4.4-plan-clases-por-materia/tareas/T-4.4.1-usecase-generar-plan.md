# T-4.4.1: Usecase GenerateClassPlan

**Historia:** HU-4.4 — Plan de clases por disciplina
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Implementar el usecase `GenerateClassPlan` en `src/core/usecases/coordination/generate_content.go`. Este usecase itera sobre cada disciplina del documento de coordinacion y envia una solicitud a Azure OpenAI con el contexto necesario para generar un plan de clases automatico. El LLM devuelve un array de objetos con la estructura de cada clase, que luego se persiste en la base de datos.

## Implementacion

### Ubicacion

`src/core/usecases/coordination/generate_content.go`

### Logica principal

1. Recibir `document_id` como parametro.
2. Cargar el documento con sus relaciones: `coord_doc_subjects`, `coord_doc_subject_topics`, secciones (`sections` JSONB).
3. Para cada disciplina en el documento:
   - Obtener: nombre de la disciplina, topics asignados (con descripciones), `class_count`, contenido de las secciones del documento.
   - Construir el prompt con este contexto (ver T-4.4.2).
   - Enviar a Azure OpenAI (`gpt-5-mini`).
   - Parsear la respuesta JSON: `[{class_number, title, objective, topic_ids}]`.
   - Validar que `topic_ids` existan en `coord_doc_subject_topics`.
4. **Si es regeneracion**: eliminar `coord_doc_classes` y `coord_doc_class_topics` existentes para esa disciplina antes de insertar los nuevos.
5. Crear registros en `coord_doc_classes` y `coord_doc_class_topics`.
6. Todo dentro de una **transaccion** para garantizar consistencia.

### Pseudocodigo

```go
func (uc *GenerateContentUsecase) GenerateClassPlan(ctx context.Context, docID int) error {
    return uc.txManager.WithTransaction(ctx, func(tx Transaction) error {
        doc, err := uc.docRepo.GetWithPreloads(ctx, tx, docID)
        if err != nil {
            return err
        }

        for _, subject := range doc.Subjects {
            // Eliminar clases existentes (regeneracion)
            err := uc.classRepo.DeleteByDocSubject(ctx, tx, docID, subject.SubjectID)
            if err != nil {
                return err
            }

            prompt := buildClassPlanPrompt(doc, subject)
            response, err := uc.aiClient.GenerateJSON(ctx, prompt)
            if err != nil {
                return err
            }

            classes, err := parseClassPlanResponse(response)
            if err != nil {
                return err
            }

            for _, class := range classes {
                classID, err := uc.classRepo.Create(ctx, tx, class)
                if err != nil {
                    return err
                }
                for _, topicID := range class.TopicIDs {
                    err := uc.classTopicRepo.Create(ctx, tx, classID, topicID)
                    if err != nil {
                        return err
                    }
                }
            }
        }
        return nil
    })
}
```

### Endpoint que invoca este usecase

`POST /api/v1/coordination-documents/:id/generate` (ya existente o definido en otra tarea).

## Notas

- Usar `max_completion_tokens` en la llamada a Azure OpenAI (requerido por `gpt-5-mini`, no soporta `max_tokens`).
- La transaccion debe cubrir la eliminacion de clases anteriores y la insercion de las nuevas para evitar estados inconsistentes.
- Si el LLM devuelve `topic_ids` invalidos, loguear el error y retornar un mensaje claro al usuario.
- Depende de T-4.4.2 para la definicion del prompt y schema de respuesta.
