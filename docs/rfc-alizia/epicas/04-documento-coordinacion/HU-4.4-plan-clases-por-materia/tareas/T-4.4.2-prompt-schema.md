# T-4.4.2: Prompt y schema para generacion de plan de clases

**Historia:** HU-4.4 — Plan de clases por disciplina
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Definir el system prompt y el JSON schema que se envia a Azure OpenAI para la generacion del plan de clases por disciplina. El prompt debe proveer suficiente contexto pedagogico y estructural para que el LLM genere un plan coherente. El schema de salida debe ser estricto para facilitar el parsing y la validacion.

## Implementacion

### Ubicacion

`src/core/usecases/coordination/prompts.go` (o similar)

### System prompt

El prompt debe incluir las siguientes secciones de contexto:

1. **Area**: nombre del area a la que pertenece la disciplina.
2. **Disciplina**: nombre de la disciplina para la cual se genera el plan.
3. **Topics asignados**: lista de topics con su ID y descripcion, tomados de `coord_doc_subject_topics`.
4. **Cantidad de clases**: `class_count` definido para esa disciplina en el documento.
5. **Secciones del documento**: contenido de las secciones relevantes (eje problematico, estrategia metodologica, etc.).

```go
const classPlanSystemPrompt = `Eres un asistente pedagogico experto en planificacion educativa.
Tu tarea es generar un plan de clases para la disciplina "{{subject_name}}" del area "{{area_name}}".

CONTEXTO DEL DOCUMENTO:
{{#each sections}}
- {{section_name}}: {{section_content}}
{{/each}}

TOPICS ASIGNADOS A ESTA DISCIPLINA:
{{#each topics}}
- ID: {{id}} | {{name}}: {{description}}
{{/each}}

INSTRUCCIONES:
- Genera exactamente {{class_count}} clases.
- Cada clase debe tener un titulo descriptivo y un objetivo pedagogico claro.
- Distribuye TODOS los topics en al menos una clase. Un topic puede aparecer en multiples clases.
- Los topic_ids deben ser IDs validos de la lista proporcionada.
- Responde UNICAMENTE con el JSON, sin texto adicional.`
```

### JSON schema de salida

```json
{
  "type": "array",
  "items": {
    "type": "object",
    "required": ["class_number", "title", "objective", "topic_ids"],
    "properties": {
      "class_number": {
        "type": "integer",
        "description": "Numero de clase, de 1 a class_count"
      },
      "title": {
        "type": "string",
        "description": "Titulo descriptivo de la clase"
      },
      "objective": {
        "type": "string",
        "description": "Objetivo pedagogico de la clase"
      },
      "topic_ids": {
        "type": "array",
        "items": { "type": "integer" },
        "description": "IDs de topics cubiertos en esta clase"
      }
    },
    "additionalProperties": false
  }
}
```

### Validacion post-respuesta

```go
func validateClassPlanResponse(classes []ClassPlanItem, validTopicIDs []int, expectedCount int) error {
    if len(classes) != expectedCount {
        return fmt.Errorf("expected %d classes, got %d", expectedCount, len(classes))
    }

    validSet := toSet(validTopicIDs)
    coveredTopics := make(map[int]bool)

    for _, class := range classes {
        for _, tid := range class.TopicIDs {
            if !validSet[tid] {
                return fmt.Errorf("topic_id %d not in coord_doc_subject_topics", tid)
            }
            coveredTopics[tid] = true
        }
    }

    // Verificar que todos los topics estan cubiertos
    for _, tid := range validTopicIDs {
        if !coveredTopics[tid] {
            return fmt.Errorf("topic_id %d not assigned to any class", tid)
        }
    }

    return nil
}
```

## Notas

- Usar `response_format: { type: "json_object" }` en la llamada a Azure OpenAI para forzar respuesta JSON valida.
- Si la validacion falla (topics faltantes o invalidos), se puede reintentar una vez antes de devolver error al usuario.
- El schema se puede enviar como parte del system prompt o como `functions` schema segun la version de la API.
- Mantener el prompt conciso pero con suficiente contexto para que el LLM genere planes pedagogicamente coherentes.
