# T-4.5.2: Usecases StartEditing y DeleteDocument

**Historia:** HU-4.5 — Publicacion y estados
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Implementar dos usecases para completar el ciclo de vida de los documentos de coordinacion:

1. **StartEditing**: transiciona un documento pendiente a en progreso. Indica que el coordinador comenzó a trabajar en el documento.
2. **DeleteDocument**: elimina permanentemente un documento, pero solo si esta en estado pendiente. Los documentos en progreso o publicados no se pueden eliminar.

## Implementacion

### Ubicacion

`src/core/usecases/coordination/archive_document.go`

---

### StartEditing

Transicion: `pending` -> `in_progress`

```go
func (uc *StartEditingUsecase) Execute(ctx context.Context, docID int) error {
    doc, err := uc.docRepo.GetByID(ctx, docID)
    if err != nil {
        return err
    }

    if doc.Status != "pending" {
        return NewValidationError("solo documentos pendientes pueden comenzar edición")
    }

    return uc.docRepo.UpdateStatus(ctx, docID, "in_progress")
}
```

**Endpoint**: Se invoca desde `PATCH /api/v1/coordination-documents/:id` cuando el body incluye `{"status": "in_progress"}`.

**Response exitosa**: `200 OK` con el documento actualizado.

---

### DeleteDocument

Solo permite eliminar documentos en estado `pending`. Realiza un hard delete con CASCADE en todas las tablas de junction asociadas.

```go
func (uc *DeleteDocumentUsecase) Execute(ctx context.Context, docID int) error {
    doc, err := uc.docRepo.GetByID(ctx, docID)
    if err != nil {
        return err
    }

    if doc.Status != "pending" {
        return NewForbiddenError("solo documentos pendientes pueden ser eliminados")
    }

    return uc.docRepo.HardDelete(ctx, docID)
}
```

**Tablas afectadas por CASCADE**:

```
coordination_documents (eliminado)
  ├── coord_doc_topics (CASCADE)
  ├── coord_doc_subjects (CASCADE)
  │     ├── coord_doc_subject_topics (CASCADE)
  │     └── coord_doc_classes (CASCADE)
  │           └── coord_doc_class_topics (CASCADE)
  └── (sections en JSONB, se elimina con el registro principal)
```

**Endpoint**: `DELETE /api/v1/coordination-documents/:id`

**Responses**:

- `204 No Content` — eliminacion exitosa
- `403 Forbidden` — documento no esta en pending

```json
{
  "error": "forbidden",
  "message": "Solo documentos pendientes pueden ser eliminados"
}
```

- `404 Not Found` — documento no existe

### Registro en router

En `mapping.go`, los endpoints ya existentes manejan estos casos:

```go
// PATCH para cambio de estado (publicar, archivar)
docs.PATCH("/:id", handler.PatchCoordinationDocument, RequireRole("coordinator"))

// DELETE para eliminacion
docs.DELETE("/:id", handler.DeleteCoordinationDocument, RequireRole("coordinator"))
```

### Logica en el handler PATCH para estados

```go
func (h *Handler) PatchCoordinationDocument(c echo.Context) error {

        case "published":

        default:
            return echo.NewHTTPError(422, "invalid status transition")
        }
    }

}
```

## Notas

- Las foreign keys en las tablas de junction deben tener `ON DELETE CASCADE` definido en las migraciones para que el hard delete funcione correctamente.
- No se implementa soft delete en esta POC para mantener la simplicidad.
- La transicion de estados valida es unidireccional: `pending` -> `in_progress` -> `published`. No hay camino de retorno.
- Un documento publicado es de solo lectura. Para modificarlo debe coordinarse con el equipo técnico.
