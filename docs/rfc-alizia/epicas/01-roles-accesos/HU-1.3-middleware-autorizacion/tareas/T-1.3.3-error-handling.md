# T-1.3.3: Error handling unificado para auth/authz

**Historia:** HU-1.3 — Middleware de autorización
**Tipo:** Backend
**Estado:** ✅ Completada

---

## Descripción

Unificar los error responses de autenticación (401) y autorización (403) para que sigan el mismo formato y sean consumibles por el frontend.

## Formato de error

```json
{
    "error": "error_code",
    "message": "Human-readable description"
}
```

## Códigos de error

| HTTP | error code | Causa |
|------|-----------|-------|
| 401 | `missing_token` | No hay header Authorization |
| 401 | `invalid_token` | JWT malformado, expirado o firma inválida |
| 403 | `forbidden` | Token válido pero rol insuficiente |
| 403 | `forbidden` | Claims presentes pero sin roles |

## Implementación

```go
// internal/middleware/errors.go
package middleware

import "team-ai-toolkit/web"

var (
    ErrMissingToken = web.ErrorResponse(401, "missing_token", "Authorization header is required")
    ErrInvalidToken = web.ErrorResponse(401, "invalid_token", "Token is invalid or expired")
    ErrForbidden    = web.ErrorResponse(403, "forbidden", "Insufficient permissions")
)
```

## Notas

- Los errores 401 vienen del JWT middleware (HU-1.1) — aquí solo los documentamos para consistencia
- Los errores 403 vienen del RequireRole middleware (T-1.3.1)
- El frontend debe manejar 401 → redirect a login, 403 → mostrar mensaje de permisos
- Usar `team-ai-toolkit/errors.HandleError()` si el error viene de una capa inferior
