# T-7.3.4: Endpoints de notificaciones

**Historia:** HU-7.3 — Sistema de notificaciones
**Tipo:** Backend
**Estado:** ⬜ Pendiente
**Fase:** Post-MVP

---

## Descripción

Implementar los endpoints REST para consultar, leer y gestionar notificaciones del usuario autenticado.

## Implementación

### Endpoints

| Método | Ruta | Descripción |
|--------|------|-------------|
| GET | `/notifications` | Listar notificaciones del usuario |
| GET | `/notifications/unread-count` | Conteo de no leídas (para badge) |
| PATCH | `/notifications/{id}/read` | Marcar una como leída |
| POST | `/notifications/read-all` | Marcar todas como leídas |

```
GET /notifications?limit=20&offset=0&unread_only=false
```

### Conteo de no leídas

```
GET /notifications/unread-count
```

```json
{
    "unread_count": 8
}
```

Este endpoint se llama al cargar cualquier página (para actualizar el badge de la campana).

### Marcar como leída

```
PATCH /notifications/{id}/read
```

```json
{
    "read_at": "2026-03-27T14:00:00Z"
}
```

### Marcar todas como leídas

```
POST /notifications/read-all
```

```json
{
    "updated_count": 8
}
```

### Queries

```sql
-- Listar notificaciones del usuario (paginadas, más recientes primero)
SELECT * FROM notifications
WHERE user_id = $1
ORDER BY created_at DESC
LIMIT $2 OFFSET $3;

-- Conteo de no leídas
SELECT COUNT(*) FROM notifications
WHERE user_id = $1 AND read_at IS NULL;

-- Marcar todas como leídas
UPDATE notifications
SET read_at = NOW()
WHERE user_id = $1 AND read_at IS NULL;
```

## Notas

- El usuario solo ve sus propias notificaciones (filtrado por user_id del token)
- `unread-count` debe ser rápido — usa el índice parcial `WHERE read_at IS NULL`
- Paginación por offset es suficiente dado el volumen esperado (< 1000 notificaciones por usuario)
- El filtro `unread_only=true` permite ver solo las pendientes
- Las notificaciones se ordenan por `created_at DESC` (más recientes primero)
